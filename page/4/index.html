【1开发】数据库动态路由组件记录1 - 路由组件

分类于 开发 ， 项目 ， mysql动态路由组件

参考学习：Apache ShardingSphere

思路

动态路由组件的一个核心思路，就是分库和分表，所以我们核心要实现如下两件事

  • 分库：根据字段属性，动态切换数据源（改数据源）
  • 分表：根据字段属性，拦截修改sql语句（改from后面的那个表）

所以可以简单设计动态路由的流程：

  1. 根据字段计算hash值，存入threadlocal
  2. 修改sql语句
  3. 切换数据源

流程1 初始化加载配置

重写springboot中EnvironmentAware的setEnvironment方法

通过environment.getProperty（）方法，我们可以直接拿到yml中的配置，也就是我们对于分库分表的配置信息

实现EnvironmentAware，就可以拿上面的这些信息了

流程2 AOP切面 通过注解 获取入参字段，并进行路由计算

既然要进行分库分表，我们首先要把用来分库分表的字段，以及其对应的值拿到手。

那么第一步，用注解拿到对应的参数，并拿到对应的值。

复习一下注解

  • interface

    • Retention(RetentionPolicy.RUNTIME) 运行时的注解

    • Target({ElementType.METHOD,ElementType.TYPE}) 修饰在方法或着类型上

    • String key() default “”; 注解的参数例如：DBRouter(key=”userId”)

  • Aspect - JoinPoint

    • Pointcut 切点 （标记对应的注解）
    • @Around(“aopPoint() && @annotation(dbRouter)”) （在该切点处运行）
    • Around方法内些具体要做的事（拿字段参数，和对应的值）

拿到了值就可以调用doRouter函数，进行hash散列的计算。

计算好的hash值存入threadlocal中

流程3 替换表名 （ibatis.plugin.Interceptor）

到了这个流程我们可以开始进行分表的操作。

我们其实是用mybatis plugin去对sql语句做一个拦截。

然后我们重写Interceptor中的intercept函数，拦截sql语句并修改

  1. invocation.getTarget().getBoundSql()拿到sql语句
  2. 利用Pattern进行替换sql语句中对应的表名
  3. 通过反射修改SQL语句

流程4 切换数据源 （jdbc.datasource.lookup.AbstractRoutingDataSource）

根据yml里拿到的那几个数据源，使用AbstractRoutingDataSource替换默认数据源

重写determineCurrentLookupKey，AbstractRoutingDataSource能够帮助我们根据threadlocal里的值动态切换数据源

【1开发】营销抽奖系统开发记录2 - DDD vs MVC

分类于 开发 ， 项目 ， 营销抽奖

贫血模型和充血模型

关于DDD和MVC的讨论，总是离不开这两个模型。在软件开发中，尤其是

贫血模型

数据访问层主要包含数据，而不包含业务逻辑（也就是只做getter/setter）。业务逻辑只在service层中出现。

充血模型

将业务逻辑和数据操作放在同一层处理，这种设计模式可以更好地处理复杂的业务逻辑和数据操作，因为业务逻辑和数据操作在同一层处理，可以减少代码的冗余和提高代码的复用性。

为什么选DDD，比起MVC好在哪里

MVC这种贫血对象腐化的根本，就是对象、服务、组件的交叉混乱使用。

MVC

首先对于MVC而言：

  1. 一个dao层操作数据库
  2. 接着是domain层里的po、vo、enum这些贫血对象（贫血，也就是只有对象，没有操作的逻辑）。
  3. 然后是service，http

会产生什么问题？

  1. 第一个service在domain里定义好了一个vo贫血对象，
  2. serivce里也经常会发生平行的调用，我要实现一个东西，发现另一个service实现好了，那么直接拿去用了
  3. 不区分业务服务和基础功能

总结一下，就是服务和服务，服务和对象之间，横向，外部，交叉怎么调用都行。很混乱，没有定义出一些约束和规范。因此能快速实现需求，但是对大项目的后期维护造成了困难。

DDD

所以，DDD相较于MVC最大的改变就是由贫血模型转变为充血模型，把你对应的仓储服务、对象、服务都写到你自己的分包模块下。你要修改，就改自己领域模块下的，不能改别人的。

【1开发】营销抽奖系统开发记录3 - Domain1_抽奖策略 库表设计

分类于 开发 ， 项目 ， 营销抽奖

抽奖策略 – 领域

“抽奖策略”领域，当用户点击抽奖按钮，后台如何计算用户该拿到什么奖品。

最简单的：不同的奖品定义了不同的概率，我们取这个区间的随机数，定位到奖品。

再复杂一点：增加抽奖规则，例如10连抽会有保底，黑号了抽不到奖

考虑抽奖前、后需要执行的策略如下

  1. 用户黑名单
  2. 用户积分权重（不同权重给到不同奖品）
  3. 库存是否满足
  4. 用户抽奖次数（某些奖品只有用户达到抽奖次数才会给）
  5. 兜底奖品（不管怎么样至少会给一点积分）

库表

Strategy

  • 策略id

Award

  • 奖品id

Strategy_rule

  • 策略id
  • 规则类型
  • 规则值

Strategy_award

  • 策略id
  • 奖品id
  • 规则树模型
  • 概率
  • 总库存
  • 剩余库存

rule_tree/rule_tree_node/rule_tree_line

  • 根节点/每个节点/每条边
  • 也就是每个节点都是一个规则，根据计算都会算出一个值，根据这个值来选择边

【1开发】营销抽奖系统开发记录4 - Domain1_抽奖策略 具体实现

分类于 开发 ， 项目 ， 营销抽奖

两种抽奖算法

时间 or 空间

第一种抽奖算法：循环对比

  • 例如：”A:0.1;B:0.2;C:0.7”

  • 计算出随机数 ：

    • 例如：0.25

  • 根据奖品的概率依次遍历，知道概率达到奖品的范围：

    • 例如：先遍历A，不在0.1；
    • 再遍历B，在(0.1+0.2=0.3)的范围内，得到B奖品

  • 时间O(n) ，空间O(1)

第二种抽奖算法：索引获取

  • 例如：”A:0.1;B:0.2;C:0.7”
  • 奖品提前放到List/Map中：【A,B,B,C,C,C,C,C,C,C】
  • 计算出随机数
  • 通过List直接拿到对应的奖品
  • 时间O(1) ，空间O(n)

选择

  • 对于抽奖这样的高并发场景，无疑是第二种索引获取更实用，保证接口的更快响应
  • 但是也有些极端场景，例如奖品概率为0.000001，这时候空间上的开销就过大了。

Service1 – 抽奖装配 （Redis预热）

全量概率

对于第二种抽奖算法，提前构造奖品概率索引Map，放入Redis中。构造的方式也很简单，根据概率的最小精度，确认Map大小，依次遍历奖品，累加概率，存入Map。

权重概率

与普通概率类似，区别在于会配置不同的权重会对应不同的奖品列表，根据指定的奖品列表，构造奖品索引概率Map，放入Redis中。

预热上图中的全量概率奖品查找表和权重概率奖品查找表

Service2 – 抽奖前置规则

主要根据用户特性进行校验

  • 黑名单
  • 积分额度大小
  • 默认抽奖规则

实现方式：责任链

也就是说，我们把所有前置规则实现（Bean），再根据mysql中策略的规则设定，将用到bean串成一个链表。

注意，这里使用单例模式是有问题的：

一开始规则实现的bean是默认的单例模式，因此动态生成责任链会有并发问题。假设策略1的规则(rule1,rule2,rule3)，bean1->bean2->bean3；后面又来了策略2的规则(rule2,rule4,rule6)，那么bean2->bean4->bean6。那么策略1的规则就被覆盖了

因此后面修改为原型模式，@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)

Service3 – 抽奖后置规则

执行完抽奖算法之后，我们拿到了奖品，需要进行抽奖后置规则计算，例如库存校验规则、抽奖次数校验规则，兜底奖励规则。

打个比方：先进行次数校验，如果满足，进入库存校验；反之，进入兜底奖励。

显然，这是一个分支结构，所以用规则树模型。

之前，我们已经定义了库表结构，所以我们首先把规则树模型结构从mysql中读进来。

实现起来，与抽奖前置规则类似，只是树形结构比起链表结构实现稍微复杂一些。

特别地，不超卖库存规则实现

“库表”这个库表属性非常，在高并发场景下，非常容易多个请求直接打到这一行上面。

比如数据库表有一条记录是库存，如果是通过锁这一条表记录更新库存为10、9、8的话，就会出现大量的用户在应用用获得数据库的连接后，等待前一个用户更新完库表记录后释放锁，让下一个用户进入在扣减。

这样随着用户参与量的增加，就会有非常多的用户处于等待状态，而等待的用户是持久数据库的连接的，这个连接资源非常宝贵，你占用了应用中其他的请求就进不来，最终导致一个请求要几分钟才能得到响应。【前台的用户越着急，越疯狂点击，直至越来越卡到崩溃】

所以，我们需要把这个扣减的动作放到缓存里

  1. decr扣减库存
  2. 使用setnx加锁兜底，防止超卖【decr是原子操作，正常情况是不会有问题的，但是可能有情况，运营手动修改库存】
  3. 同时将扣减动作放入MQ，缓慢修改mysql库存

串联抽奖规则

至此，我们就可以实现一个基本的抽奖流程

  • 输入：用户id，活动id
  • 执行抽奖（包含相关抽奖过滤）
  • 输出：奖品

API

  1. 策略装配，将策略信息装配到缓存中；/api/v1/raffle/strategy_armory
  2. 查询奖品列表；/api/v1/raffle/query_raffle_award_list
  3. 随机抽奖接口；/api/v1/raffle/random_raffle

抽奖活动 - 领域

用户抽奖的业务流程分为；给自己的活动账户添加额度（购买、兑换、打卡），领取活动（扣减互动账户额度）、执行抽奖策略、抽奖结果落库。

前两天完成了“抽奖策略”部分的领域代码，那么在“计算用户能拿到什么奖品”前，还需要干两件事

  • 管理用户的账户抽奖额度
  • 领取抽奖活动，生成抽奖活动流水单，扣减账户额度

所以“抽奖活动”领域主要做两个Service：账户额度，参与活动

抽奖活动 - 库表设计

每个抽奖活动次数可以看成一个sku（活动id，策略id，库存），同时还需要定义用户的活动账号，以及订单库表

分库分表

对于用户账号信息，只分库；对于用户订单信息，分库分表

Service1 参与活动

参与活动可以看作商品sku下单，只是这个sku是活动次数。

所以，现在主要实现两个service

  1. 下单规则校验（责任链设计模式实现）
  2. 完成下单事务（这个单是抽奖活动次数sku）

Service1-接口1 下单规则校验

校验库存，活动是否在有效期内，活动是否开启，采用责任链模式（不过这个比较简单，可能也用不着责任链模式），和之前的抽奖前置规则一样

Service1-接口2 用户点击抽奖按钮，创建抽奖订单（更新额度，写入用户抽奖订单）

更新 raffle_activity_account（账户抽奖次数）

更新 user_raffle_order（用户抽奖订单）

  • step1 构建订单聚合对象
  • step2 事务

Service2 抽奖活动账户额度

用户通过某种行为（签到、积分），换取抽奖次数，那么主要干两件事

Service2-接口1 扣减活动“库存”

设计到库存，和之前的奖品库存扣减类似，用缓存进行decr，用setnx兜底，写入MQ趋势更新库存

Service2-接口2 每日签到送抽奖次数额度（后续加上积分）

【1开发】营销抽奖系统开发记录8 - Domain5_积分

分类于 开发 ， 项目 ， 营销抽奖

积分 - 领域

积分在本项目中取代钱的作用，积分是衔接营销场景中各个功能模块的一个非常重要的手段，主要是打卡获得积分，抽奖获得积分，积分兑换抽奖次数sku。

核心接口1 交易策略

这里的交易只有一种商品sku（抽奖额度sku）

涉及到两种交易策略

  • 签到赠送积分 – 无支付 （较为简单，直接写入订单，修改抽奖额度，就结束了）
  • 积分兑换 – 有支付 （较为复杂，除了写入订单，修改抽奖额度，还需要考虑积分账户的扣减）

核心接口2 订单出货

更新订单状态操作，是对订单从支付态更改为发货完成态。在数据库表更新的时候，有一个前置的判断，订单状态必须为 wait_pay 才能更新为 completed 否则不能更新。

核心接口3 交易支付

交易支付这部分，需要与发货解耦，不在交易中直接对另外的库表进行一个事务的处理。因为如果将来积分要拆分为独立系统的时候，太重的事务耦合就不好拆分了，而用 MQ 解耦流程，就可以很方便的完成系统模块的拆分。

监听积分支付成功消息后，使用发货单实体对象为入参，调用订单变更操作，完成积分充值。
