MySQL加索引需要注意些什么

MySQL事务的隔离级别

读未提交，读已提交（MVCC，每次读取生成一个readview），可重复读（MVCC，第一次读取的时候生成一个readview），串行化（加锁）。

【开发】动态线程池组件记录1 - 线程池调研

分类于 开发 ， 项目 ， 动态线程池组件

ThreadPoolExecutor

学习资料：Java线程池实现原理及其在美团业务中的实践

基本概念

线程池（是什么）

池化思想管理线程，好处

  • 降低资源消耗：重复利用已创建的线程
  • 提高响应速度：任务到达时，无需等待线程创建
  • 提高线程的可管理性：线程是稀缺资源，使用线程进行统一的分配、调优和监控
  • 提供更多更强大的功能：ScheduledThreadPoolExecutor

线程池解决的问题（为什么）

线程是稀缺资源，不能够确定在任意时刻中，有多少任务需要执行，有多少资源需要投入

  • 频繁申请/销毁资源，带来额外的消耗
  • 对资源无限申请，易引发系统资源耗尽的风险

池化思想：

  • 内存池
  • 连接池
  • 实例池

线程池核心设计与实现

总体设计

  • 顶层接口Exector：
    • 任务提交和任务执行解藕
    • 用户只需要将Runnable提交到Executor中
  • ExecutorService：
    • 补充执行任务的能力，生成Future
    • 管控线程池，停止线程池的运行
  • AbstractExecutorService：
    • 抽象类，串联执行任务的流程
  • ThreadPoolExecutor
    • 实现类

线程池内部实际上构建了一个生产者消费者模型，将线程和任务两者解藕，缓冲任务，复用线程。

任务管理（生产者）

  • 直接申请线程执行任务
  • 缓冲队列中等待线程执行
  • 拒绝该任务

线程管理（消费者）

  • 它们被统⼀维护在线程池内，根据任务请求进⾏线程的分配

生命周期管理

维护两个值：运行状态（runState），工作线程数量（workerCount）

运行状态有五种：

  • RUNNING：正常运行
  • SHUTDOWN：不再接收新任务，能继续处理
  • STOP：不接受新任务，中断线程
  • TIDYING：
  • TERMINATED

任务执行机制

任务调度

  • 先看核心线程有没有满
  • 再看阻塞队列
  • 最后看最大线程数
  • 都不行，就拒绝

任务缓冲（阻塞队列）

任务申请

⼀种是任务直接由新创建的线程执⾏。

另⼀种是线程从任务队列中获取任务然后执⾏，执⾏完任务的空闲线程会再次去从队列中申请任务再去执⾏。

任务拒绝

任务队列满，线程池满，采取任务拒绝策略

  • 丢弃任务，抛出异常
  • 丢弃任务，不抛出异常
  • 丢弃最前面的任务，提交被拒绝的任务
  • 由调用线程处理该任务

Worker线程管理

thread是在调⽤构造⽅法时通过ThreadFactory来创建的线程，可以⽤来执⾏任务；firstTask⽤它来保存传⼊的第⼀个任务，这个任务可以有也可以为null。如果这个值是⾮空的，那么线程就会在启动初期⽴即执⾏这个任务，也就对应核⼼线程创建时的情况；如果这个值是null，那么就需要创建⼀个线程去执⾏任务列表（workQueue）中的任务，也就是⾮核⼼线程的创建。

Worker集成AQS，独占锁，比如回收场景

  • Lock方法一旦获取了锁，代表该线程在执行任务，不能中断回收
  • 不是独占，空闲，可以回收

Worker线程执行任务

  1. while循环不断getTask()方法来获取任务
    1. 如果getTask为null则跳出循环，销毁线程
  2. 判断线程池是否stop
  3. 执行任务

实践场景

1 - 快速响应用户请求

描述

实时请求，⽐如说⽤户要查看⼀个商品的信息，那么我们需要将商品维度的⼀系列信息如商品的价格、优惠、库存、图⽚等等聚合起来，展示给⽤户。

分析

不设置队列去缓冲并发任务，调高corePoolSize和maxPoolSize去尽可能创造多的线程快速执⾏任务。

2 - 快速处理批量任务

描述

离线任务，需要快速执行，统计报表

分析

不需要瞬时完成，吞吐量优先。设置队列去缓冲并发任务，调整合适的corePoolSize去设置处理任务的线程数。在这⾥，设置的线程数过多可能还会引发线程上下⽂切换频繁的问题，也会降低处理任务的速度，降低吞吐量。

思路

动态路由组件的一个核心思路，就是分库和分表，所以我们核心要实现如下两件事

  • 分库：根据字段属性，动态切换数据源（改数据源）
  • 分表：根据字段属性，拦截修改sql语句（改from后面的那个表）

所以可以简单设计动态路由的流程：

  1. 根据字段计算hash值，存入threadlocal
  2. 修改sql语句
  3. 切换数据源

流程1 初始化加载配置

重写springboot中EnvironmentAware的setEnvironment方法

通过environment.getProperty（）方法，我们可以直接拿到yml中的配置，也就是我们对于分库分表的配置信息

实现EnvironmentAware，就可以拿上面的这些信息了

流程2 AOP切面 通过注解 获取入参字段，并进行路由计算

既然要进行分库分表，我们首先要把用来分库分表的字段，以及其对应的值拿到手。

那么第一步，用注解拿到对应的参数，并拿到对应的值。

复习一下注解

  • interface

    • Retention(RetentionPolicy.RUNTIME) 运行时的注解

    • Target({ElementType.METHOD,ElementType.TYPE}) 修饰在方法或着类型上

    • String key() default “”; 注解的参数例如：DBRouter(key=”userId”)

  • Aspect - JoinPoint

    • Pointcut 切点 （标记对应的注解）
    • @Around(“aopPoint() && @annotation(dbRouter)”) （在该切点处运行）
    • Around方法内些具体要做的事（拿字段参数，和对应的值）

拿到了值就可以调用doRouter函数，进行hash散列的计算。

计算好的hash值存入threadlocal中

流程3 替换表名 （ibatis.plugin.Interceptor）

到了这个流程我们可以开始进行分表的操作。

我们其实是用mybatis plugin去对sql语句做一个拦截。

然后我们重写Interceptor中的intercept函数，拦截sql语句并修改

  1. invocation.getTarget().getBoundSql()拿到sql语句
  2. 利用Pattern进行替换sql语句中对应的表名
  3. 通过反射修改SQL语句

流程4 切换数据源 （jdbc.datasource.lookup.AbstractRoutingDataSource）

根据yml里拿到的那几个数据源，使用AbstractRoutingDataSource替换默认数据源

核心技术：SpringBoot、MyBatis、MySQL、Redis、Kafka、React

项目描述：该项目是我参考字节掘金中幸运抽奖模块开发的一个幸运大转盘抽奖系统。该系统支持多种营销策略，如：用户积分，积分兑换抽奖次数，返利等；也支持各类差异化抽奖流程，如：消耗积分指定抽奖范围、抽奖N次解锁奖品等各类抽奖玩法。该系统采用DDD架构进行设计，将整个系统划分为抽奖活动、抽奖策略、奖品、积分、返利这五个领域。同时设计中运用到了责任链模式、组合模式、工厂模式等多种设计模式，提高了代码的可扩展性，并对抽奖的计算和秒杀做了设计的优化，可以支撑单机 4c8g 服务器 1100 TPS 的吞吐量，抽奖接口响应时长160毫秒左右。

核心职责

  • 采用策略模式实现两种抽奖算法，一种是总体概率，空间复杂度为O(1)；一种是单项概率，将对应概率的概率的奖品提前映射到数组中，时间复杂度为O(1)。

  • 根据不同的抽奖玩法，设计整体抽奖策略。在抽奖流程的抽象上，分为抽奖前和抽奖后，如抽奖前的人群判断、抽奖后兜底奖励等。设计模版模式定义抽奖流程，抽奖前调用责任链完成抽奖，抽奖后的动作使用组合模式的规则树进行动态处理。

  • 抽奖是一种峰值流量高的业务场景，针对抽奖活动库存和奖品库存的扣减：采用了 Redis decr 分段消费和加锁兜底的设计，同时对于消费成功的库存，异步队列方式 + 定时任务更新库存。当库存扣减为0时，采用MQ对数据库库存清零并清空队列。减轻数据库压力并确保数据最终一致性。

  • 针对打卡返利、积分兑换活动、抽奖下发奖品等流程，采用MQ，将积分支付和活动账户充值、抽奖和奖品下发等流程，进行解藕。并且通过task表进行补偿，确保消息成功消费。

  • 对于用户账户、用户活动订单、积分返利订单等库表，采取分库分表的方式

【开发】营销抽奖系统开发记录2 - DDD vs MVC

分类于 开发 ， 项目 ， 营销抽奖

贫血模型和充血模型

关于DDD和MVC的讨论，总是离不开这两个模型。在软件开发中，尤其是

贫血模型

数据访问层主要包含数据，而不包含业务逻辑（也就是只做getter/setter）。业务逻辑只在service层中出现。

充血模型

将业务逻辑和数据操作放在同一层处理，这种设计模式可以更好地处理复杂的业务逻辑和数据操作，因为业务逻辑和数据操作在同一层处理，可以减少代码的冗余和提高代码的复用性。

为什么选DDD，比起MVC好在哪里

下图，展示了MVC这种贫血对象腐化的根本，就是对象、服务、组件的交叉混乱使用。

MVC

首先对于MVC而言：

  1. 一个dao层操作数据库
  2. 接着是domain层里的po、vo、enum这些贫血对象（贫血，也就是只有对象，没有操作的逻辑）。
  3. 然后是service，http

会产生什么问题？

  1. 第一个service在domain里定义好了一个vo贫血对象，
  2. serivce里也经常会发生平行的调用，我要实现一个东西，发现另一个service实现好了，那么直接拿去用了
  3. 不区分业务服务和基础功能

总结一下，就是服务和服务，服务和对象之间，横向，外部，交叉怎么调用都行。很混乱，没有定义出一些约束和规范。因此能快速实现需求，但是对大项目的后期维护造成了困难。

DDD

所以，DDD相较于MVC最大的改变就是由贫血模型转变为充血模型，把你对应的仓储服务、对象、服务都写到你自己的分包模块下。你要修改，就改自己领域模块下的，不能改别人的。

抽奖策略 – 领域

“抽奖策略”领域，当用户点击抽奖按钮，后台如何计算用户该拿到什么奖品。

最简单的：不同的奖品定义了不同的概率，我们取这个区间的随机数，定位到奖品。

再复杂一点：增加抽奖规则，例如10连抽会有保底，黑号了抽不到奖

考虑抽奖前、后需要执行的策略如下

  1. 用户黑名单
  2. 用户积分权重（不同权重给到不同奖品）
  3. 库存是否满足
  4. 用户抽奖次数（某些奖品只有用户达到抽奖次数才会给）
  5. 兜底奖品（不管怎么样至少会给一点积分）

【开发】营销抽奖系统开发记录4 - Domain1_抽奖策略 具体实现

分类于 开发 ， 项目 ， 营销抽奖

两种抽奖算法

时间 or 空间

第一种抽奖算法：循环对比

  • 例如：”A:0.1;B:0.2;C:0.7”

  • 计算出随机数 ：

    • 例如：0.25

  • 根据奖品的概率依次遍历，知道概率达到奖品的范围：

    • 例如：先遍历A，不在0.1；
    • 再遍历B，在(0.1+0.2=0.3)的范围内，得到B奖品

  • 时间O(n) ，空间O(1)

第二种抽奖算法：索引获取

  • 例如：”A:0.1;B:0.2;C:0.7”
  • 奖品提前放到List/Map中：【A,B,B,C,C,C,C,C,C,C】
  • 计算出随机数
  • 通过List直接拿到对应的奖品
  • 时间O(1) ，空间O(n)

选择

  • 对于抽奖这样的高并发场景，无疑是第二种索引获取更实用，保证接口的更快响应
  • 但是也有些极端场景，例如奖品概率为0.000001，这时候空间上的开销就过大了。

Service1 – 抽奖装配 （Redis预热）

全量概率

对于第二种抽奖算法，提前构造奖品概率索引Map，放入Redis中。构造的方式也很简单，根据概率的最小精度，确认Map大小，依次遍历奖品，累加概率，存入Map。

权重概率

与普通概率类似，区别在于会配置不同的权重会对应不同的奖品列表，根据指定的奖品列表，构造奖品索引概率Map，放入Redis中。

预热上图中的全量概率奖品查找表和权重概率奖品查找表

Service2 – 抽奖前置规则

主要根据用户特性进行校验

  • 黑名单
  • 积分额度大小
  • 默认抽奖规则

实现方式：责任链

也就是说，我们把所有前置规则实现（Bean），再根据mysql中策略的规则设定，将用到bean串成一个链表。

注意，这里使用单例模式是有问题的：

一开始规则实现的bean是默认的单例模式，因此动态生成责任链会有并发问题。假设策略1的规则(rule1,rule2,rule3)，bean1->bean2->bean3；后面又来了策略2的规则(rule2,rule4,rule6)，那么bean2->bean4->bean6。那么策略1的规则就被覆盖了

因此后面修改为原型模式，@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)

Service3 – 抽奖后置规则

执行完抽奖算法之后，我们拿到了奖品，需要进行抽奖后置规则计算，例如库存校验规则、抽奖次数校验规则，兜底奖励规则。

打个比方：先进行次数校验，如果满足，进入库存校验；反之，进入兜底奖励。

显然，这是一个分支结构，所以用规则树模型。

之前，我们已经定义了库表结构，所以我们首先把规则树模型结构从mysql中读进来。

实现起来，与抽奖前置规则类似，只是树形结构比起链表结构实现稍微复杂一些。

特别地，不超卖库存规则实现

“库表”这个库表属性非常，在高并发场景下，非常容易多个请求直接打到这一行上面。

比如数据库表有一条记录是库存，如果是通过锁这一条表记录更新库存为10、9、8的话，就会出现大量的用户在应用用获得数据库的连接后，等待前一个用户更新完库表记录后释放锁，让下一个用户进入在扣减。

这样随着用户参与量的增加，就会有非常多的用户处于等待状态，而等待的用户是持久数据库的连接的，这个连接资源非常宝贵，你占用了应用中其他的请求就进不来，最终导致一个请求要几分钟才能得到响应。【前台的用户越着急，越疯狂点击，直至越来越卡到崩溃】

所以，我们需要把这个扣减的动作放到缓存里

  1. decr扣减库存
  2. 使用setnx加锁兜底，防止超卖【decr是原子操作，正常情况是不会有问题的，但是可能有情况，运营手动修改库存】
  3. 同时将扣减动作放入MQ，缓慢修改mysql库存
